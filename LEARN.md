## 第一章：先搞懂这个系统在现实中是什么

想象一个场景：

> 某私募基金的量化模型，每天早上9点15分自动算出今天要买哪些股票、卖哪些股票，生成一张"指令清单"。
> 然后有一个Python程序，把这张清单发给券商（比如华泰、中信），券商收到后在9点30分开盘时执行。

这就是我们做的东西。

```
量化模型 → 生成 signals.csv → Python客户端 → 发送给券商服务器 → 券商执行买卖
```

## 第二章：三个文件各自干什么

### 📄 signals.csv — 交易指令清单

```csv
date,time,stock_code,direction,action,volume,price,signal_id
2026-04-28,09:30:00,000001.SZ,BUY,OPEN,1000,15.50,SIG_20260428_001
```

**逐列解释：**

| 列名 | 意思 | 举例 |
|------|------|------|
| `stock_code` | 股票代码，.SZ=深圳，.SH=上海 | 000001.SZ = 平安银行 |
| `direction` | 买还是卖 | BUY / SELL |
| `action` | 开仓（新买入）还是平仓（卖出已持有的） | OPEN / CLOSE |
| `volume` | 买多少股，必须是100的倍数（1手=100股） | 1000 |
| `price` | 限价，0表示市价（不限价，立刻成交） | 15.50 |
| `signal_id` | 这条指令的唯一编号，**最重要** | SIG_20260428_001 |

**为什么signal_id最重要？**

想象你发了一条"买1000股平安银行"的指令，网络卡了，你不确定对方收没收到，于是又发了一次。如果没有signal_id去重，你就买了2000股——这在金融系统里是灾难性错误。

有了signal_id，券商看到同一个ID，第二次直接回复"已处理，忽略"，永远只执行一次。这叫**幂等性**。

### 🖥️ mock_broker.py — 假装是券商的服务器

真实券商（华泰、中信）有自己的服务器，你不能随便连。所以我们自己建一个假的，行为完全一样。

**它做三件事：**

**① 接收并验证信号**
```python
class TradeSignal(BaseModel):
    volume: int
    
    @field_validator("volume")
    def check_volume(cls, v):
        if v % 100 != 0:
            raise ValueError("A股最小100股")
        return v
```
Pydantic负责检查数据格式。如果你发来的volume是150（不是100的倍数），直接拒绝，不进入下一步。就像海关查护照，格式不对不让进。

**② 幂等性检查**
```python
if signal.signal_id in received_signals:
    return {"status": "duplicate"}  # 重复，忽略
```
内存里记录所有已处理的signal_id。见过的直接说"重复"，不再执行。

**③ 随机模拟失败**
```python
if random.random() < 0.1:
    raise HTTPException(status_code=503)  # 10%概率假装宕机
```
真实券商高峰期会偶尔不可用。我们故意让它10%概率失败，逼迫客户端学会处理错误。

### 🤖 upload_client.py — 你的信号发送程序

**它的完整工作流：**

```
1. 打开 signals.csv，逐行读取
        ↓
2. 把每行数据打包成字典
        ↓
3. POST请求发给券商的 /api/v1/signals/upload
        ↓
   ┌─ 成功？→ 记录订单号，继续下一条
   ├─ 503？→ 等待，重试（最多3次）
   └─ 连接失败？→ 报错，停止
        ↓
4. 全部处理完，查询账户状态
        ↓
5. 把结果写到 logs/ 文件夹
```

## 第三章：最关键的两个设计——亲眼看到的

### 🔁 指数退避（Exponential Backoff）

你运行时看到过这个：
```
11:09:03 [WARNING]  ⚠️  券商暂时不可用，2秒后重试...
11:09:05 [INFO]     ✅ 成功
```

代码是这样的：
```python
for attempt in range(1, MAX_RETRIES + 1):
    try:
        response = requests.post(...)
        return response.json()
    except HTTPError as e:
        if e.response.status_code == 503:
            wait = 2 ** attempt   # 第1次等2秒，第2次等4秒，第3次等8秒
            time.sleep(wait)
```

**为什么不是每次等固定时间？**

如果失败了每隔1秒重试，系统宕机时所有客户端同时重试，会把服务器打垮（雪崩）。指数退避让等待时间越来越长，给服务器恢复的时间。

---

### ♻️ 幂等性保护（你踩到过的坑）

你遇到过这个输出：
```
📊 上传完成: ✅成功 0  ♻️重复 5  ❌失败 0
```

**为什么全是重复？**

服务器没重启，内存里还存着上次的5条signal_id。第二次来的全部被认为是重复。

**这个Bug教会你的：**
- 开发环境用内存存信号ID，轻量方便
- 生产环境必须用数据库（PostgreSQL）持久化，否则服务器重启就丢状态

```
开发环境：received_signals = {}  ← 字典，重启清空
生产环境：INSERT INTO signal_log(signal_id, ...) ON CONFLICT DO NOTHING  ← 数据库，永久保存
```

## 第四章：日志系统——为什么要写到文件

你运行时看到最后两行：
```
📁 结果已保存: logs/result_20260429_111711.json
📁 日志已保存: logs/upload_20260429_111711.log
```

**为什么不只打印在终端？**

| 只打印终端 | 写到文件 |
|-----------|---------|
| 关了终端就没了 | 永久保存 |
| 不能搜索 | 可以grep查关键词 |
| 出了事没证据 | 可以追溯"哪条信号什么时候失败的" |

**JSON文件是对账用的：**
```json
{
  "summary": {"success": 4, "duplicate": 1, "failed": 0},
  "details": [
    {"signal_id": "SIG_001", "status": "accepted", "order_id": "ORD_52C50020"},
    {"signal_id": "SIG_001", "status": "duplicate"}
  ]
}
```
每天收盘后，用这个文件跟券商的成交记录核对，查是否有信号丢失或重复执行。

## 第五章：整个系统的思维体系

### 任何"客户端发请求给服务器"的系统，都要考虑这四个问题：

```
┌─────────────────────────────────────────────────────┐
│  1. 数据格式对不对？                                  │
│     → Pydantic校验：错误数据在入口拦截，不进业务逻辑   │
│                                                       │
│  2. 网络失败了怎么办？                                 │
│     → 重试 + 指数退避：不能立刻放弃，也不能无脑狂发    │
│                                                       │
│  3. 重试会不会重复执行？                               │
│     → 幂等性：用唯一ID去重，同一个操作只执行一次        │
│                                                       │
│  4. 出了问题怎么查？                                   │
│     → 结构化日志：每条操作记录时间、结果、订单号        │
└─────────────────────────────────────────────────────┘
```

这四个问题不只是量化交易系统要考虑——**任何需要可靠性的系统都适用**：
- 支付系统（重复扣款 = 幂等性问题）
- 外卖下单（网络超时 = 重试问题）
- 数据同步（格式错误 = 校验问题）
- 故障排查（日志不完整 = 可观测性问题）

## 一句话总结

> 你今天亲手做的事情，本质上是：**把数据可靠地从A送到B，同时处理好"送不到"和"送重了"这两个问题。**
> 
> 这是所有分布式系统、金融系统、API开发的核心挑战。你从一个具体的量化场景里理解了它——比背100个概念都管用。
