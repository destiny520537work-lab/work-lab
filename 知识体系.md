## 第一章：服务器是什么？

### 🍕 外卖比喻

> 你饿了，打开美团，点了一份披萨。
> 
> 你 = 客户端（发出需求的人）  
> 美团服务器 = 服务器（接收需求、处理、返回结果的那一方）  
> 披萨 = 数据（服务器返回给你的东西）

服务器本质上就是：**一直开着、等待别人来问它要东西的程序。**

你的 `mock_broker.py` 就是一个服务器：
- 一直运行（所以那个终端不能关）
- 等待 `upload_client.py` 来发信号
- 收到信号就处理，然后回复结果

## 第二章：HTTP是什么？

### 📬 快递比喻

HTTP是客户端和服务器之间"说话"的规则，就像寄快递有固定格式：
- 收件人地址
- 物品内容
- 是寄东西（POST）还是要东西（GET）

**两种最常用的"动作"：**

| 动作 | 意思 | 生活比喻 |
|------|------|---------|
| `GET` | 我要拿数据 | 打电话问"你们营业吗？" |
| `POST` | 我要送数据给你 | 填表格提交给柜台 |

我们的系统：
```python
# 客户端发送信号（送数据给券商）
requests.post("http://localhost:8000/api/v1/signals/upload", json=signal_data)

# 客户端查账户状态（要数据）
requests.get("http://localhost:8000/api/v1/account/status")
```

## 第三章：JSON是什么？

### 📦 标准化包装盒比喻

两台电脑之间传数据，要用双方都看得懂的格式。  
JSON就是互联网上的**标准包装格式**——就像快递箱，全世界通用。

```json
{
  "signal_id": "SIG_20260428_001",
  "stock_code": "000001.SZ",
  "direction": "BUY",
  "volume": 1000,
  "price": 15.50
}
```

**规则只有几条：**
- 用 `{}` 包裹整体
- 键（左边）必须是字符串，用 `""` 引号
- 值（右边）可以是：字符串、数字、布尔值、列表、另一个JSON
- 键值之间用 `:`，每对之间用 `,`

Python里：
```python
import json

# Python字典 → JSON字符串（打包）
data = {"name": "avril", "age": 23}
json_str = json.dumps(data)   # '{"name": "avril", "age": 23}'

# JSON字符串 → Python字典（拆包）
back = json.loads(json_str)   # {"name": "avril", "age": 23}
```

## 第四章：API是什么？

### 🔌 插座比喻

API = Application Programming Interface = 程序之间的"插座"

> 你家墙上有插座，规格是220V两孔。  
> 只要你的电器符合这个规格，就能用，不用知道墙里的电线怎么走的。

券商的API就是他们对外提供的"插座"——你只要按照他们规定的格式发请求，就能让他们帮你下单，不需要知道他们内部系统怎么工作的。

**我们mock broker的三个"插座"：**

```
POST /api/v1/signals/upload    ← 发交易信号
GET  /api/v1/account/status    ← 查账户状态
GET  /api/v1/signals/received  ← 查已收到的信号
```

每个插座有固定的"规格"（输入什么格式、返回什么格式）。

## 第五章：FastAPI是什么？

### 🏗️ 建筑脚手架比喻

你自己用Python写一个服务器，需要处理很多底层问题：
- 怎么监听端口？
- 怎么解析HTTP请求？
- 怎么返回正确的格式？

FastAPI帮你把这些都做好了，你只需要专注写业务逻辑：

```python
from fastapi import FastAPI

app = FastAPI()          # 创建一个服务器

@app.get("/hello")       # 告诉它：有人来访问 /hello 时...
def say_hello():
    return {"msg": "你好"}   # ...就返回这个
```

就这几行，一个能运行的服务器就好了。

**`@app.get("/hello")` 这行叫什么？**

叫**装饰器（Decorator）**。  
意思是：把下面这个函数"装饰"成一个API接口。  
就像在门上贴了个牌子："这扇门负责处理GET /hello的请求"。

## 第六章：Pydantic是什么？

### 🛂 海关比喻

> 入境需要护照、签证、填入境卡。  
> 少一个？对不起，不让进。  
> 格式错了？重新填。

Pydantic就是API入口的海关——数据格式不对，拒之门外。

```python
from pydantic import BaseModel

class TradeSignal(BaseModel):
    stock_code: str      # 必须是字符串
    volume: int          # 必须是整数
    price: float         # 必须是小数
```

你发来 `volume: "一千"`（字符串而不是整数）？  
Pydantic直接拒绝，返回422错误，连业务代码都不会跑到。

**好处：**
- 不用自己写 `if type(volume) != int` 这种检查
- 错误信息自动生成，清晰明了
- 代码更干净

## 第七章：状态码是什么？

### 🚦 红绿灯比喻

服务器每次回应都带一个数字，告诉你"结果怎么样了"：

| 状态码 | 意思 | 生活比喻 |
|--------|------|---------|
| `200` | 成功 | 绿灯，通行 |
| `201` | 创建成功 | 绿灯，新建了一个东西 |
| `400` | 你发来的格式有问题 | 你填的表格有错 |
| `422` | 数据校验失败（Pydantic） | 海关说你材料不对 |
| `500` | 服务器内部出错 | 对方系统崩了 |
| `503` | 服务器暂时不可用 | 对方在维修，稍后再来 |

我们模拟的 `503` 就是假装券商在"维修"，逼迫客户端重试。

## 第八章：指数退避是什么？

### ⏰ 失恋比喻（别笑）

> 你发消息给对方，对方没回（503）。
>
> 笨方法：每隔1秒就发一次 → 骚扰，越发越烦  
> 聪明方法：等2秒，等4秒，等8秒，越来越长 → 给对方喘息时间

```python
wait = 2 ** attempt   # 第1次2秒，第2次4秒，第3次8秒
time.sleep(wait)
```

这就是**指数退避**——失败了不立刻放弃，但也不穷追猛打，给对方留余地。

**为什么不能每次等相同时间？**

想象券商服务器刚宕机，1000个客户端每隔1秒同时发请求——服务器刚要恢复，立刻被1000个请求打垮，又宕机了。  
指数退避让大家等待时间错开，服务器才有机会恢复。

## 第九章：幂等性是什么？

### 🔁 电梯按钮比喻

> 电梯按了一次"3楼"，没反应，你又按了一次。  
> 电梯只去一次3楼，不会因为你按了两次就去两次3楼。

这就是幂等性：**无论执行多少次，结果和执行一次完全相同。**

我们的实现：
```python
# 服务器端
if signal.signal_id in received_signals:
    return {"status": "duplicate"}   # 见过这个ID，直接说"重复"

received_signals[signal.signal_id] = signal  # 第一次才真正处理
```

**为什么金融系统必须有幂等性？**

> 你转账1000块，网络超时，你不知道对方收没收到，再转了一次。  
> 没有幂等性 → 对方收到2000块  
> 有幂等性 → 对方只收到1000块（第二笔识别为重复，拒绝）

## 第十章：日志是什么？

### 🕵️ 监控录像比喻

日志就是系统的监控录像——平时不看，出了事回放查真相。

```
11:09:03 [INFO]    ✅ 成功 | 订单号: ORD_C11694BE
11:09:03 [WARNING] ⚠️  券商暂时不可用，2秒后重试...
11:09:05 [INFO]    ✅ 成功 | 订单号: ORD_25F9E5A1
```

**日志级别：**

| 级别 | 用途 |
|------|------|
| `INFO` | 正常流程记录，"一切OK" |
| `WARNING` | 有点不对劲，但还能继续 |
| `ERROR` | 出错了，需要关注 |
| `CRITICAL` | 严重故障，系统可能崩溃 |

**为什么要写到文件而不只打印？**

终端一关，打印的内容全消失。文件会永远留着。  
出了事：`grep "ERROR" logs/upload_20260429.log` → 立刻找到所有错误。

## 🧠 总结：用一张图串起所有概念

```
你写的代码
│
├── upload_client.py（客户端）
│   ├── requests库        ← 发HTTP请求的工具
│   ├── 指数退避重试       ← 失败了不放弃，越等越久
│   ├── 日志系统          ← 记录每一步发生了什么
│   └── JSON结果文件      ← 对账用的记录
│
└── mock_broker.py（服务器）
    ├── FastAPI           ← 搭服务器的脚手架
    ├── Pydantic          ← 数据校验（海关）
    ├── 幂等性检查         ← signal_id去重，防重复下单
    └── 状态码            ← 200成功/503暂时不可用
```

**所有这些，解决的都是同一个问题：**

> 如何把数据**可靠地**从A送到B——  
> 格式对（Pydantic）、送不到时知道重试（指数退避）、重试不会重复执行（幂等性）、出了事能查（日志）。
